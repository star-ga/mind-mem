## Multi-LLM consensus (Grok-4.1, Mistral-Large, DeepSeek-Reasoner, Gemini-3.1-Pro)

### Tier 1 — 4★+ consensus, ship first

#### 1. Hybrid BM25 + BGE-large-en-v1.5 with RRF (★★★★★ 4.75)
Semantic recall complements BM25 lexical matching. RRF k=60 fusion.  
**Why it works:** LoCoMo queries use paraphrased language; BGE catches semantic matches BM25 misses. Unanimous top pick.  
**Implementation:** Already have `hybrid_recall.HybridBackend`; wire BGE embedding server on `:8090` (requires stopping Ollama temporarily to free RTX 3080 VRAM, OR use BGE-small 512d which fits alongside).

#### 2. LLM query decomposition with UNION retrieval (★★★★ 4.25)
Replace regex `NLPQueryDecomposer` with Opus few-shot decomposition. **KEY:** union the original-question retrieval with sub-query retrievals into a single top-(k×(n_subq+1)) pool. Do NOT RRF-fuse — that's what caused v3.3.0 multi-hop regression.  
**Why it works:** regex can't handle implicit multi-hop; union preserves the bridge-context the fusion dropped.  
**Implementation:** Update `query_planner.LLMQueryDecomposer` to always include `question` as sub_query[0]. Replace RRF-fuse in `hybrid_recall.search` when decomposition active with union+dedup.

#### 3. Iterative retrieval / Chain-of-Retrieval (NEW — 3/4 LLMs raised unprompted)
After initial top-k retrieval, have Opus generate 1-2 follow-up queries based on what it found and what's missing. Re-retrieve with those. Up to 2 rounds.  
**Why it works:** multi-hop "bridge" questions need second-hop triggered by first-hop evidence. Pre-decomposition guesses; iterative discovers.  
**Implementation:** new module `src/mind_mem/iterative_recall.py` wrapping `hybrid_recall.search`. 2-round cap to bound latency.

### Tier 2 — 3-4★ consensus, ship after Tier 1

#### 4. Graph expansion ±1 hop with decay=0.5 (★★★ 3.75)
Already shipped as `graph_recall.graph_expand` but NOT wired into locomo_judge. Wire it in.  
**Why it works:** LoCoMo has ref-style links; graph expand surfaces blocks missed by BM25+vector.

#### 5. Chain-of-note evidence packing (★★★ 3.75)
Have Opus summarize retrieved blocks into 3-5 bullet notes before answering.  
**Why it works:** reduces "lost in the middle" on 10-20 retrieved blocks. Mistral named this specifically.

#### 6. Temporal metadata injection (NEW — Gemini raised)
Prepend each retrieved block with `[Stored T days ago]` relative to query time. Inject reference date in prompt.  
**Why it works:** our temporal category drops to 39 on full-conv because model has no anchor. This gives it one.

#### 7. Per-category prompts (★★★ 3.75)
Already shipped as `answer_quality.prompt_for_category`. NOT wired into locomo_judge. Wire it in.

### Tier 3 — exploratory, defer

- **LLM reranker** (★★ 2.5) — marginal if BGE already good
- **Paraphrase query expansion** (★★ 2.5) — redundant with hybrid
- **Temp-sampled self-consistency** (★★ 2.25) — Opus deterministic breaks this
- **Fine-tune encoder on LoCoMo** (★★★ 3.5) — high effort, uncertain lift, skip for v3.4.0

## What v3.3.0 feature experiment taught us

Do NOT ship these without further work:
1. **Regex-only decomposition** (`NLPQueryDecomposer`) — too blunt, misses implicit multi-hop structure.
2. **RRF-fuse of sub-query results** — attenuates the seed question's best hits.
3. **Self-consistency with Opus temp=0** — deterministic outputs, 3x cost for zero lift.

These modules stay in the codebase as building blocks (the refactored interfaces are correct), but locomo_judge should NOT invoke them as-is.
