# mind-mem — Postgres/SQLite backend parity audit (2026-06-14)

**Status:** audit complete (14 real parity bugs); fixes pending.

**Dominant root cause:** the PostgresBlockStore CRUD surface is solid, but the recall / scan / governance / export / reindex / daemon FEATURE layer reads the LOCAL MARKDOWN corpus (parse_file + CORPUS_DIRS) and the local SQLite FTS index, ignoring the configured block_store backend. So a Postgres user's blocks are invisible to those features (the sqlite_only_count drift) until a non-obvious mm doctor --rebuild-cache. SQLite is the default and must stay green; Postgres + Redis are OPTIONAL and must work out-of-the-box.

## Bugs

### 1. [CRITICAL] recall pipeline never queries the configured Postgres block store — recall returns nothing for PG-stored blocks
- **Module:** src/mind_mem/_recall_core.py:1462 (_load_backend) + :428 (recall dispatch); src/mind_mem/mcp/tools/recall.py:157-211 (_recall_impl_uncached)
- **Error:** mm recall "Postgres backend" -> [] despite 2 matching blocks in mm_smoke_fresh.blocks. _load_backend() only recognises recall.backend in {scan,sqlite,vector}; it has NO awareness of block_store.backend=postgres, so it returns None and recall() falls through to the markdown-file BM25 scan (recall_engine) which scans local .md files that don't exist on a fresh PG workspace. The MCP recall/hybrid_search tools go through HybridBackend / fts_query (SQLite FTS5) / recall_engine — none of which ever call PostgresBlockStore.hybrid_search().
- **Root cause:** The recall/retrieval layer is hardwired to the local markdown corpus + local SQLite FTS index. There is no RecallBackend implementation that reads from PostgresBlockStore, and _load_backend does not bridge block_store.backend -> a PG recall path. A PG user only gets results if they (a) manually set recall.backend=sqlite AND (b) run `mm doctor --rebuild-cache` to copy PG rows into the local recall.db. Neither is auto-wired or documented in the init config.
- **Fix:** Add a PostgresRecallBackend(RecallBackend) that delegates search() to PostgresBlockStore.hybrid_search(), and make _load_backend() return it when config['block_store']['backend']=='postgres' (unless recall.backend explicitly overrides). Alternatively, have the recall dispatch in _recall_core.recall() and mcp/tools/recall.py consult get_block_store(ws) and call its search/hybrid_search when the store is non-markdown.

### 2. [HIGH] MCP recall/scan/get_block/export and all 34 ws-gated tools fail-closed or no-op on a fresh PG workspace because _check_workspace requires a local decisions/ directory
- **Module:** src/mind_mem/mcp/infra/workspace.py:62-72 (_check_workspace)
- **Error:** MCP recall("Postgres backend") -> {"error": "Workspace is missing the 'decisions/' directory. Run: mind-mem-init <path>"} even though 2 blocks are correctly stored in PG. _check_workspace() requires os.path.isdir(ws/decisions); for a PG backend the blocks live in Postgres, not in local decisions/.
- **Root cause:** _check_workspace hardcodes the markdown-corpus directory layout as the definition of a valid workspace. For a Postgres-backed workspace the canonical block storage is the DB; the local decisions/ dir is irrelevant. 34 call sites across nearly every tool module gate on this, so the whole MCP surface assumes markdown layout.
- **Fix:** Make _check_workspace backend-aware: if block_store.backend in {postgres} (or the configured store is non-markdown), validate via the store (e.g. store.ping()/list_blocks succeeds) instead of requiring decisions/ on disk. At minimum, skip the decisions/ check when block_store.backend!='markdown'.

### 3. [HIGH] scan (contradiction detection) is blind to Postgres — reports decisions.total=0 / contradictions.raw=0 with contradictory blocks in PG
- **Module:** src/mind_mem/mcp/tools/governance.py:203-257 (scan)
- **Error:** Wrote 2 contradictory decision blocks ('default backend is SQLite' vs 'default backend is Postgres') to PG; MCP scan() returns {checks:{decisions:{total:0,active:0},contradictions:{raw:0,resolvable:0}}}. scan() parse_file()s ws/decisions/DECISIONS.md and ws/intelligence/*.md only.
- **Root cause:** scan() reads the local markdown corpus directly via parse_file and never consults get_block_store(). On the PG backend the governance/contradiction feature silently does nothing — a correctness gap, not just slow.
- **Fix:** Route scan()'s block enumeration through get_block_store(ws).get_all(active_only=True) (and conflict_resolver/contradiction_detector likewise) so contradiction/drift detection sees the chosen backend's blocks. Keep markdown parse_file as the fallback only for the markdown backend.

### 4. [HIGH] MCP reindex (build_index) indexes only local markdown — does NOT sync PG, and actively introduces sqlite_only_count drift
- **Module:** src/mind_mem/sqlite_index.py:696-750 (build_index) + src/mind_mem/mcp/tools/memory_ops.py:146 (reindex)
- **Error:** PG has 3 active blocks; after MCP reindex(), index_stats reports total_blocks(FTS)=1 and `mm doctor` flips to sqlite_only_count=1, pg_only_count=3, in_sync=false. build_index() iterates CORPUS_FILES and parse_file()s local markdown, ignoring the PostgresBlockStore. It indexed a markdown template block that never existed in PG (the source of sqlite_only_count=1).
- **Root cause:** There are two divergent index-build paths: build_index() (markdown-only, used by the reindex tool and reindex_dirty) and the PG-aware `mm doctor --rebuild-cache` path in mm_cli.py:1749. The standard `reindex` tool an agent would call is markdown-only, so on PG it both fails to import PG rows and injects markdown-only rows into the recall cache (the exact sqlite_only_count drift the spec warns about).
- **Fix:** Make build_index()/reindex backend-aware: when block_store.backend=='postgres', source rows from get_block_store(ws).get_all() (the same logic as doctor --rebuild-cache) instead of parse_file over CORPUS_FILES, or unify both into one store-driven indexer.

### 5. [MEDIUM] export_memory / get_block / find_similar / memory_evolution read local markdown, missing PG blocks
- **Module:** src/mind_mem/mcp/tools/memory_ops.py:321 (export_memory), :385 (get_block), and CORPUS_DIRS loops at :96/:338/:403/:422/:478
- **Error:** PG holds D-20260613-001 (store.get_by_id returns it) but MCP get_block('D-20260613-001') -> {found:false, error:'Block ... not found in any corpus file.'}. export_memory returns 0 exported blocks with 2-3 blocks in PG.
- **Root cause:** memory_ops tools enumerate blocks via parse_file over CORPUS_DIRS rather than get_block_store(ws), so on the PG backend they cannot see the blocks of record.
- **Fix:** Switch the block enumeration in export_memory/get_block/find_similar/memory_evolution to get_block_store(ws).get_all()/get_by_id() when a non-markdown backend is configured.

### 6. [MEDIUM] pgvector embedding column silently disabled (has_vector=False) when the DSN search_path drops 'public' (the schema-isolation pattern the audit itself recommends)
- **Module:** src/mind_mem/block_store_postgres.py:204-220 (_try_create_extension_vector) + :223-257/:471-484 (_ddl_pgvector / _ensure_schema)
- **Error:** Log 'postgres_pgvector_ddl_failed'; _ddl_pgvector raised UndefinedObject: type "vector" does not exist on `ADD COLUMN ... VECTOR(1024)`. CREATE EXTENSION IF NOT EXISTS vector returns success (extension already exists in public) but the unqualified VECTOR type can't resolve because the DSN used ?options=-csearch_path=mm_smoke_fresh (no public). Result: has_vector=False, hybrid_search degrades to bm25_only with no error surfaced to the user. With the documented 'schema' kwarg + default search_path (public present) has_vector=True correctly.
- **Root cause:** _try_create_extension_vector() reports success based on CREATE EXTENSION (idempotent no-op when it already exists elsewhere) but _ddl_pgvector references the VECTOR type unqualified, assuming the extension's schema is on search_path. When a user isolates the workspace by overriding search_path to just their schema (a common/recommended isolation), public drops off and the type is unreachable.
- **Fix:** Detect the extension's actual namespace (SELECT extnamespace::regnamespace FROM pg_extension WHERE extname='vector') and either schema-qualify the type ({ext_schema}.vector) in _ddl_pgvector, or CREATE EXTENSION ... WITH SCHEMA / ensure the extension schema is appended to search_path before the ALTER. Also have _try_create_extension_vector actually verify the type is resolvable (e.g. SELECT 'x'::vector roundtrip) rather than trusting CREATE EXTENSION's return.

### 7. [HIGH] daemon scheduled jobs (intel_scan, entity_ingest, transcript_scan) crash: package modules invoked as bare script paths -> relative-import ImportError
- **Module:** src/mind_mem/cron_runner.py:99 (cmd=[sys.executable, script_path, workspace]); scripts src/mind_mem/intel_scan.py:25, entity_ingest.py, transcript_capture.py
- **Error:** cron_runner job_failed: "File .../intel_scan.py, line 25, from .block_parser import parse_file -> ImportError: attempted relative import with no known parent package". Reproduced directly: `python3 src/mind_mem/intel_scan.py <ws>` raises the same. All 3 job scripts use `from .` relative imports.
- **Root cause:** run_job() builds cmd = [sys.executable, <abs path to module>.py, ...] and runs it as a top-level script, but intel_scan/entity_ingest/transcript_capture are package modules using relative imports, which only resolve under `python -m mind_mem.<mod>`. The background daemon's scheduled jobs therefore never run (errors on every backend, including PG).
- **Fix:** Change run_job to invoke via the module runner: cmd = [sys.executable, '-m', f'mind_mem.{module_name}', workspace, *extra_args] (map JOB_DEFS to module names rather than file paths). Add a smoke test that each daemon job exits 0 on an init'd workspace.

### 8. [MEDIUM] init_workspace has no Postgres support: generated mind-mem.json omits block_store and no --backend/--dsn flags; also creates a junk './--help' directory
- **Module:** src/mind_mem/init_workspace.py:88-95 (config template) + arg parsing (treats --help as workspace path)
- **Error:** python3 -m mind_mem.init_workspace <ws> writes a config with only {recall:{backend:'bm25'}} and NO block_store section; there is no flag to select postgres. `--help` is consumed as a workspace path and created a literal directory /home/n/mind-mem/--help (removed during audit).
- **Root cause:** The documented first-run (mind-mem-init) only scaffolds the SQLite/markdown layout. A Postgres user must hand-edit mind-mem.json to add the block_store {backend,dsn,schema} block after init — there is no out-of-box path to a working PG install, and the init CLI lacks argparse (so --help/--backend aren't handled).
- **Fix:** Add argparse to init_workspace with --backend {markdown,postgres,encrypted} and --dsn/--schema; when postgres is chosen, write the block_store section (and set recall.backend appropriately) and optionally call store._ensure_schema(). Reject unknown flags instead of treating them as the workspace path.

### 9. [CRITICAL] recall returns 0 results on Postgres: FTS indexer reads markdown corpus, never the PG block store
- **Module:** src/mind_mem/sqlite_index.py:696 build_index (reads CORPUS_FILES via parse_file lines 721-725); src/mind_mem/_recall_core.py:1462 _load_backend (no 'postgres' backend; only scan/sqlite/vector); recall path at _recall_core.py:666 parse_file
- **Error:** Blocks written to PostgresBlockStore persist in PG (verified: 'PG blocks written'; psql shows D-301/D-302). build_index(WS, incremental=False) -> {'files_checked':9,'files_indexed':9,'blocks_indexed':0,'total_blocks':0}. recall(WS,'Postgres') -> 0 results. Direct PostgresBlockStore.hybrid_search('Postgres') -> 2 hits (D-301,D-302) — so data IS queryable in PG, the recall layer just never asks PG. No error raised; silent empty result with message 'No matching blocks found.'
- **Root cause:** The recall/FTS pipeline is markdown-centric. build_index and the BM25 scan path read decisions/DECISIONS.md, tasks/TASKS.md, etc. via parse_file. On the Postgres backend those corpus files are the empty init templates (all real blocks live in the PG blocks table). _load_backend only recognises scan/sqlite/vector — there is no recall backend that reads from PostgresBlockStore. So a brand-new PG user who writes blocks then calls recall gets 0 results. PostgresBlockStore even implements hybrid_search() but the recall layer never calls it. The only sync path is the non-obvious `mm doctor --rebuild-cache`, which copies pg_only blocks into recall.db + blocks_fts (verified: after rebuild, recall('Postgres') -> 1).
- **Fix:** Make the recall FTS source backend-aware. Option A: in build_index/reindex, when block_store.backend=='postgres', enumerate blocks via get_block_store(ws).list_blocks()/get_by_id (or read the PG blocks table) instead of (or in addition to) parse_file over CORPUS_FILES, so the FTS cache mirrors PG. Option B: add a 'postgres' RecallBackend in _load_backend that delegates to PostgresBlockStore.hybrid_search/search so recall queries PG directly. Option C (stopgap): auto-run the doctor rebuild-cache sync on first recall when pg_only>0. Preferred: A or B so recall works out-of-box on PG with no manual doctor step.

### 10. [HIGH] scan (contradiction detection) is a silent no-op on Postgres — reads markdown corpus, not the block store
- **Module:** src/mind_mem/mcp/tools/governance.py:203 scan() (parse_file decisions/DECISIONS.md line 214); src/mind_mem/contradiction_detector.py:265 detect_contradictions (iterates COMMITTED_CORPUS_FILES markdown, parse_file line 270)
- **Error:** Seeded 2 contradictory blocks into PG (D-201 'use Postgres as the only backend' / D-202 'will not use Postgres; SQLite is the only backend'), both confirmed in PG via psql. gov.scan() -> {'checks':{'decisions':{'total':0,'active':0},'contradictions':{'raw':0,'resolvable':0},'drift_items':0}}. - **Root cause:** scan() and contradiction_detector.detect_contradictions read the markdown corpus files (decisions/DECISIONS.md, tasks/TASKS.md, MEMORY.md ...) directly via parse_file. On the PG backend these files are empty init templates, so the contradiction/governance scan sees no blocks and silently reports a clean workspace even when PG contains contradictory blocks.
- **Fix:** Route the corpus enumeration in scan()/detect_contradictions through get_block_store(ws): when backend is postgres (or generically), load blocks from the store (list_blocks + get_by_id) instead of parse_file over on-disk markdown. Factor a shared _load_corpus_blocks(ws) helper that all governance/maintenance passes (scan, dream_cycle, drift_detector) call so the source-of-blocks is backend-aware in one place.

### 13. [MEDIUM] index_stats crashes with OperationalError 'attempt to write a readonly database' when recall.db exists with only the calibration table
- **Module:** src/mind_mem/sqlite_index.py:1288 index_status (conn = _connect(workspace, readonly=True); then _init_schema(conn) at line 1288 executes CREATE TABLE DDL — a write — at sqlite_index.py:181). Surfaced via src/mind_mem/mcp/tools/memory_ops.py:134 index_stats (fts_status call)
- **Error:** mm/MCP index_stats -> {'error':'database backend error','error_type':'OperationalError','tool':'index_stats'}. Raw: sqlite_index.py:181 _init_schema conn.executescript(...) -> sqlite3.OperationalError: attempt to write a readonly database. Repro: CalibrationManager(ws).record_feedback() creates .mind-mem-index/recall.db with only [calibration_feedback, sqlite_sequence] tables; then index_status() sees the file exists, opens it mode=ro, and _init_schema tries CREATE TABLE on the read-only handle -> crash.
- **Root cause:** index_status assumes 'recall.db file exists => its schema (blocks/meta/blocks_fts) exists', so it opens read-only then calls _init_schema. But calibration.py (and other features sharing recall.db) can create recall.db first with only their own tables. On the PG backend the FTS schema is frequently never built (blocks live in PG), so recall.db commonly contains only the calibration table — making this crash reliable on PG. The OperationalError 'readonly' is also not the 'database is locked' case the index_stats handler special-cases, so it escapes to the generic db-error path.
- **Fix:** In sqlite_index.index_status, do not call _init_schema on a readonly connection: either (a) open writable to ensure schema then reopen ro, or (b) detect missing tables via sqlite_master and return {'exists':True,'blocks':0,...} without attempting DDL, or (c) guard with 'CREATE TABLE IF NOT EXISTS' on a writable connection only when the schema is absent. Also ensure all writers of recall.db (calibration, retrieval_graph, sqlite_index) initialise the full shared schema, or give calibration its own DB file so recall.db is never half-initialised.

### 14. [HIGH] index_status() calls _init_schema() (CREATE TABLE) on a mode=ro connection → OperationalError 'attempt to write a readonly database' on default SQLite
- **Module:** src/mind_mem/sqlite_index.py:1287-1288 (index_status), root in _connect readonly=True at line 106-108; surfaced/mis-handled in src/mind_mem/mcp/tools/memory_ops.py:78-90 (index_stats)
- **Error:** RAW from index_status(ws): OperationalError -> attempt to write a readonly database. Through the MCP wrapper: {"error": "database backend error", "error_type": "OperationalError", "tool": "index_stats", "_schema_version": "1.0"} and logged event mcp_tool_db_error error='attempt to write a readonly database'. Fully reproducible on pure default SQLite, zero PG/Redis.
- **Root cause:** index_status() is documented read-only and opens the DB with `_connect(workspace, readonly=True)` (URI mode=ro). It then unconditionally calls `_init_schema(conn)`, which runs `CREATE TABLE IF NOT EXISTS ...`. When recall.db exists but lacks the index schema (the `blocks`/`meta` tables) — the normal state right after `recall`/`hybrid_search` lazily create recall.db via the calibration + retrieval_log side-tables (calibration.py shares the SAME `.mind-mem-index/recall.db` per _DB_REL_PATH) but BEFORE `reindex`/`build_index` runs — the CREATE TABLE on a mode=ro connection fails. index_stats() only catches OperationalError that `_is_db_locked()` recognises (memory_ops.py:87-90) and re-raises everything else, so the user gets a generic 'database backend error'. Sequence: recall (partial recall.db) -> index_stats -> error; after reindex it works (total_blocks correct, error None).
- **Fix:** index_status() should not write on a readonly handle. Either (a) skip _init_schema when readonly and instead detect a missing schema by querying sqlite_master (SELECT name FROM sqlite_master WHERE type='table' AND name IN ('blocks','meta')) and return {exists:True, blocks:0, stale_files:...} when the index tables are absent; or (b) wrap the _init_schema/SELECT in a try/except sqlite3.OperationalError that treats 'no such table'/'readonly database' as 'index not built yet'. Also broaden index_stats() (memory_ops.py:87-90) to treat non-locked OperationalError as fts_exists=False (fall through to the parse-files branch) instead of re-raising. This keeps the default zero-config path from erroring before the first reindex.
