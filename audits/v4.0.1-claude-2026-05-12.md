## Security findings

### S-1 — Non-constant-time token compare 🔴 Critical
**Where**: `src/mind_mem/http_transport.py:640`
**What**: `_authenticated()` uses `sent == token` (plain Python equality,
short-circuits on first mismatch). The MCP path at `mcp/infra/http_auth.py:117`
correctly uses `hmac.compare_digest` — the two auth paths are inconsistent.
**Why it matters**: A local-network attacker can brute-force the token
character-by-character by timing latency, eventually reaching `POST /clear`
and `DELETE /memories/{id}`.
**Fix**: `import hmac; return bool(sent) and hmac.compare_digest(sent, token)`

### S-2 — No rate-limiting on HTTP transport 🟠 High
**Where**: `src/mind_mem/http_transport.py` (entire file)
**What**: Source inspection confirms zero rate-limiter calls in `do_GET` /
`do_POST` / `do_DELETE`. The `SlidingWindowRateLimiter` in
`mcp/infra/rate_limit.py` is only reachable via the `@mcp_tool_observe`
decorator — unreachable from the HTTP transport.
**Why it matters**: `POST /consolidate` (full dream cycle), `POST /query`
(vector search), `POST /clear` callable in a tight loop by any
authenticated caller.
**Fix**: Instantiate `SlidingWindowRateLimiter` at `serve_http()` and call
`limiter.allow()` at the top of each handler.

### S-3 — Federation 503 leaks internal exception text 🟡 Medium
**Where**: `src/mind_mem/http_transport.py:493, 511, 551, 599`
**What**: All four `_handle_fed_*` functions catch `Exception` and return
`f"federation disabled: {exc}"`. `FileNotFoundError` surfaces absolute
workspace path; `sqlite3.OperationalError` surfaces schema details;
`ImportError` surfaces internal module structure.
**Fix**: Separate `ImportError`/`FeatureDisabledError` from generic; log
unexpected internally; return `"federation unavailable"` without detail.

### S-4 — `GET /status` leaks absolute workspace path 🟡 Medium
**Where**: `src/mind_mem/http_transport.py:198`
**What**: `os.path.abspath(workspace)` in the 200 body — applies even to
unauthenticated-localhost responses.
**Fix**: Return only `os.path.basename(workspace)`, or expose via a
dedicated admin endpoint.

### S-5 — `/federation/vclock/<block_id>` missing path-traversal guard 🟡 Medium
**Where**: `src/mind_mem/http_transport.py:679-683` (and 541-548, 580-597)
**What**: `_handle_delete_memory` (line 373) guards `block_id` against `/`
and `..`; the federation handlers do not. SQL is parameterized so no SQLi,
but `block_id` is echoed back into responses and any future file-keyed
lookup inherits raw user input.
**Fix**: Apply the same `if "/" in block_id or ".." in block_id or len(block_id) > 256` guard.

### S-6 — arch_mind MCP tools pass unsanitized paths to subprocess 🟠 High
**Where**: `src/mind_mem/mcp/tools/arch_mind.py:71-116, 131, 144, 170-172, 190-200`
**What**: `repo`, `fixture`, `before`, `after`, `rules`, `agent_id`,
`commit_sha` passed verbatim into `subprocess.run()` arg list. `shell=False`
prevents OS-shell injection but flag injection to the underlying
`arch-mind` binary is unguarded. These tools sit in `USER_TOOLS`, so no
admin token required to reach them.
**Fix**: Validate path args via `_validate_path` (reuse from `workspace.py`).
Allowlist `mode` to `{"enforce", "report"}`. Enforce
`agent_id`/`commit_sha` against `^[a-zA-Z0-9_-]{1,128}$`.

### S-7 — No CORS/CSRF guard on HTTP transport 🟡 Medium
**Where**: `src/mind_mem/http_transport.py` (no `do_OPTIONS`, no Origin check)
**What**: Custom header (`X-MindMem-Token`) blocks simple CSRF only if
pre-flight `OPTIONS` is rejected. There's no `Origin`-header guard.
**Fix**: `do_OPTIONS → 405`; reject non-`http://127.0.0.1`/`localhost` Origin.

### S-8 — `MIND_MEM_ACL_DISABLED` permanently disables admin ACLs 🟠 High
**Where**: `src/mind_mem/mcp/infra/observability.py:89-96`
**What**: When the env var is truthy, the ACL check is skipped for every
tool — including `delete_memory_item`, `decrypt_file`, `encrypt_file`,
`rollback_proposal`. A one-time warning is emitted then silenced by
`_acl_disabled_warned`; subsequent admin-tool calls through user tokens
are never logged as bypasses.
**Fix**: Remove from production code (test-only flag), and remove the
one-shot warning suppression so every bypassed call is logged.

### S-9 — `--token` CLI flag exposes secret in `/proc/cmdline` 🟡 Medium
**Where**: `src/mind_mem/mcp/server.py:209-214`
**What**: `main()` warns about cmdline exposure but then continues. Token
visible in `/proc/<pid>/cmdline` for server lifetime — readable by any
same-user process and captured by audit frameworks.
**Fix**: Remove the flag; require `MIND_MEM_TOKEN` env var only.

### S-10 — TOCTOU symlink race in `_safe_vault_path` / `_validate_path` 🟠 High
**Where**: `src/mind_mem/mcp/tools/encryption.py:37-38`,
`src/mind_mem/mcp/infra/workspace.py:80-81`
**What**: `os.path.realpath()` resolves symlinks at validation time, but
the symlink can be replaced between validation and the subsequent
`open()`. An attacker controlling a writable directory in the workspace
can race the validate→open window and have `encrypt_file` open an
arbitrary out-of-workspace target (e.g., SSH keys).
**Fix**: Reject symlinks outright at validation time; use
`os.open(path, os.O_WRONLY | os.O_NOFOLLOW)` for writes.

### S-11 — `deleted_blocks.jsonl` recovery log written without file lock 🟢 Low
**Where**: `src/mind_mem/mcp/tools/memory_ops.py:270-278`
**What**: `delete_memory_item` holds a `FileLock` on the source `.md` file
but appends to `memory/deleted_blocks.jsonl` with no lock. The
`ThreadingMixIn` server dispatches concurrently — two deletions can
interleave their jsonl writes, corrupting the recovery log.
**Fix**: Add a second `FileLock` on `deleted_blocks.jsonl`, or `flock` the fd.

### Confirmed defenses
- ✅ SQL parameterized everywhere in `v4/federation.py`,
  `federation_client.py`, `http_transport.py` — no f-string / `%` / `.format()`
  in any `conn.execute()` call.
- ✅ Body cap `MAX_BODY_BYTES = 1 MiB` enforced in `_read_body()` before
  base64 decode; effective post-decode size ≤ ~768 KB (S-3 caveat aside).
- ✅ `_safe_log()` strips CR/LF/NUL from user-controlled log values.
- ✅ MCP path uses `hmac.compare_digest`.

## Federation v-clock + TAG_v1 preimage findings

### FP-1 — LAST_WRITER_WINS ≡ HIGHER_VERSION 🟠 High
**Where**: `src/mind_mem/v4/federation.py:312-315`
**What**: Both enum arms fall into the same branch and pick
`report.left_agent` (line 266 sorts agents by *version* descending, so
"left" is always the highest-version agent). The module docstring (line 14)
says LAST_WRITER_WINS = "Highest agent version wins" — making it
identical to HIGHER_VERSION. The `resolve_conflict` docstring (line 292)
contradicts: "Pick whichever side wrote most recently (by version vector
tip)" — a wall-clock semantic. Neither strategy reads `last_seen_at` even
though the schema stores it (line 126).
**Fix**: For LAST_WRITER_WINS, query `block_tier_vclock.last_seen_at` and
pick the row with the more recent timestamp (break ties with version).
Or collapse the two enum values into one if the distinction is dropped.

### FP-2 — `_log_conflict` check-then-act TOCTOU 🟠 High
**Where**: `src/mind_mem/v4/federation.py:354-381`
**What**: The duplicate-check `SELECT` and the `INSERT` are in a plain
context — no `BEGIN IMMEDIATE`. No `UNIQUE` constraint on
`(block_id, left_agent, right_agent, left_version, right_version)`. Two
threads calling `detect_conflict` simultaneously will both evaluate
`existing is None` and both insert.
**Fix**: Add `conn.execute("BEGIN IMMEDIATE")` before the SELECT, OR add a
partial `UNIQUE` index filtered on `resolution IS NULL` and use
`INSERT OR IGNORE`.

### FP-3 — `resolve_conflict` UPDATE over-broad 🟠 High
**Where**: `src/mind_mem/v4/federation.py:327-330`
```sql
UPDATE tier_conflict_log SET resolution=?, resolved_to=?, resolved_at=?
WHERE block_id=? AND resolution IS NULL
```
**What**: No filter on `left_agent`, `right_agent`, `left_version`,
`right_version`. If two open rows exist for the same `block_id` (different
version pairs), both close with the same strategy and winner — silently
falsifying the stale row's audit metadata.
**Fix**: `_log_conflict` returns the inserted `rowid`; `resolve_conflict`
pins UPDATE to `WHERE rowid = ?`.

### FP-4 — THREE_WAY_MERGE with `merger=None` returns None silently 🟡 Medium
**Where**: `src/mind_mem/v4/federation.py:316-318`
**What**: Three distinct outcomes all return None: (1) no conflict found,
(2) strategy=THREE_WAY_MERGE but merger=None (caller error), (3)
unrecognized strategy. Public docstring documents only (1).
**Fix**: Raise `ValueError("THREE_WAY_MERGE requires a merger callable")`.

### FP-5 — `preimage()` None semantics: doc/type-alias/impl all disagree 🟡 Medium
**Where**: `src/mind_mem/preimage.py:24-26`, `:55-56`, `:43`
**What**: Module docstring claims `None → empty bytes`. `_render` raises
`ValueError`. Type alias `Value = Union[..., None]` still includes None,
so mypy accepts a None call that always raises at runtime.
**Fix**: Drop None from the `Value` union. Rewrite docstring lines 24-26
to state None is rejected; require `""` or sentinel.

### FP-6 — `+Inf` and any `float >= 32768.0` produce identical preimages 🟡 Medium
**Where**: `src/mind_mem/q1616.py:46-48`; `src/mind_mem/preimage.py:69-73`
**What**: `to_q16_16` saturates `+Inf` to `_MAX = 0x7FFFFFFF`. Any finite
float that overflows Q16.16 also saturates to `_MAX`. Empirically:
`hex_q16_16(float('inf')) == hex_q16_16(32768.0) == '7fffffff'` — they
collide in preimage. Preimage rejects NaN but lets `+Inf` through.
**Fix**: Add `math.isinf(value)` guard in `_render` alongside the NaN
check: `raise ValueError("preimage(): ±Inf is not hashable")`.

### FP-7 — Three+ agent divergence: only top-2 logged 🟡 Medium
**Where**: `src/mind_mem/v4/federation.py:265-267`
**What**: `detect_conflict` picks `sorted_agents[0]` + `sorted_agents[1]`
only. With agents A:10, B:7, C:3, only A vs B is logged; C is invisible
indefinitely. After A vs B resolves, the same top-2 pair re-appears and C
is still ignored.
**Fix**: Iterate `sorted_agents[1:]` and log one row per lagging agent.

### FP-8 — `_log_conflict` + `resolve_conflict` missing `BEGIN IMMEDIATE` 🟡 Medium
**Where**: `src/mind_mem/v4/federation.py:354, 326`
**What**: `record_agent_write` uses `BEGIN IMMEDIATE` (correct). The
log/update writes don't. Two concurrent `resolve_conflict` calls can both
succeed; the second silently overwrites the first's audit metadata.
**Fix**: Add `conn.execute("BEGIN IMMEDIATE")` in both paths.

### FP-9 — Legacy v1 audit-chain `|` separator: pipe in target path 🟢 Low
**Where**: `src/mind_mem/audit_chain.py:157`
**What**: Linux paths can contain `|`; a target like `"docs/a|b.md"`
shifts field boundaries and causes verify() to flag a false tamper. The
v3 downgrade-attack mitigation at line 378 limits the blast radius.
**Fix**: Emit `LEGACY_SEPARATOR_AMBIGUITY` distinct from generic tamper so
operators can distinguish encoding artifact from real tampering.

### Confirmed preimage correctness
- ✅ `bool` vs `int` distinctness verified (`True → b"t"`, `1 → b"1"`).
- ✅ NaN rejection correct (`value != value` is the right IEEE 754 test).
- ✅ Q16.16 negative-hex encoding correct (`q & 0xFFFFFFFF` mask works on
  all CPython targets).
- ✅ Cross-arch float stability defensible in the stated `[0.0, 1.0]` domain.
- ✅ Downgrade-attack mitigation correctly blocks retroactive v3→v1 hash
  downgrade after a v3 entry is seen.

## Retrieval + RRF fusion findings

### R-1 — RRF first-seen metadata wins (silent staleness) 🟠 High
**Where**: `src/mind_mem/hybrid_recall.py:99-100`
**What**: `block_data[bid]` is set only on first encounter. If BM25
(list 0) returns a stale-metadata snapshot and vector (list 1) returns
the same block with fresher metadata, BM25's version is permanently kept.
RRF score is correct; every field comes from whichever list is first.
**Fix**: Prefer the higher-`Date` metadata in dedup, or take a union with
the fresher field winning.

### R-2 — Weight normalization undocumented; `rrf_score` inflates with large weights 🟡 Medium
**Where**: `src/mind_mem/hybrid_recall.py:95`, docstring 72-86
**What**: Weights are relative multipliers, not probability masses. Setting
`vector_weight=10.0` inflates all `rrf_score` values ~5x. These scores
flow to `knee_cutoff` (`_recall_core.py:163-204`) as absolute thresholds.
**Fix**: Either normalize (`w /= sum(weights)`), or document explicitly
and add a validation warning for extreme values.

### R-3 — Partial-empty `ranked_lists` silent degradation 🟡 Medium
**Where**: `src/mind_mem/hybrid_recall.py:88-110`, metric at `:411`
**What**: When BM25 succeeds + vector fails *at runtime*, `ranked_lists =
[bm25_results, []]`. RRF math is mathematically correct (empty list
contributes zero), but score values are roughly half. The
`hybrid_searches_fused` counter still increments — single-leg degradation
is invisible.
**Fix**: After both futures complete, emit
`hybrid_single_list_degenerate` metric.

### R-4 — `_get_block_id` `file:line` fallback silent dedup collision 🟡 Medium
**Where**: `src/mind_mem/hybrid_recall.py:113-124`
**What**: Two distinct chunks of the same source can share `file:line`
and collapse in the first-seen dedup with no log.
**Fix**: `_log.warning("rrf_fallback_id_used", ...)` when fallback is taken.

### R-5 — Multi-query expansion runs serially, not in parallel 🟡 Medium
**Where**: `src/mind_mem/hybrid_recall.py:503-515`
**What**: `for q in queries` calls `self.search()` serially. Each call
spawns its own 2-thread pool internally, but the N expansion variants wait
for each other. With N=4 at ~50ms each, total ~200ms instead of ~50ms.
`prefetch_context` correctly uses `as_completed` for the analogous pattern.
**Fix**: Replace serial loop with `ThreadPoolExecutor(max_workers=len(queries))`.

### R-6 — `detect_query_type` called 3× per request 🟢 Low
**Where**: `src/mind_mem/hybrid_recall.py:257, 312, 760`
**What**: Same regex classification on the same query, three times. Not
memoized.
**Fix**: Compute once at top of `search()`, or `functools.lru_cache`.

### R-7 — `rebuild_index` bypasses provider chain + writes wrong format 🟠 High
**Where**: `src/mind_mem/recall_vector.py:1371, 1377-1388`
**What**: Calls `backend.model.encode(...)` directly — bypasses the
Ollama → llama_cpp → fastembed → sentence-transformers fallback chain AND
the circuit-breaker. Writes embeddings in the legacy flat `index.json`
format which the sqlite-vec search path never consults. A workspace
configured with `provider=sqlite_vec` calling `rebuild_index` silently
gets no usable index.
**Fix**: Replace with `backend._embed_for_provider(texts)`; route to the
correct provider-specific index format based on `self.provider`.

### R-8 — sqlite-vec `active_only` 3× overfetch not adaptive 🟢 Low
**Where**: `src/mind_mem/recall_vector.py:727`
**What**: `fetch_limit = limit * 3 if active_only else limit`. In sparse-
active workspaces (<33% active), filtered result count < limit. Vector
leg silently contributes fewer candidates to RRF than intended.
**Fix**: If `len(results) < limit // 2`, log warning + retry with doubled
`fetch_limit`, or expose an `overfetch_factor` config key.

### R-9 — Pinecone missing key → silent BM25-only degradation 🟡 Medium
**Where**: `src/mind_mem/recall_vector.py:1221-1223`,
`src/mind_mem/hybrid_recall.py:881-883`
**What**: `_search_pinecone` raises `ValueError("PINECONE_API_KEY
required")`. Caught by bare `except Exception` in `_vector_search`, returns
`[]`. Hybrid continues BM25-only with HTTP 200, `hybrid_searches_fused`
still increments.
**Fix**: At `VectorBackend.__init__`, when `provider == "pinecone"`, check
for the key immediately and log a WARNING. In `_vector_search`, re-raise
or emit CRITICAL when the error message contains `"API_KEY"`.

### R-10 — `_maybe_temporal_decay` mutates fused dicts in-place 🟢 Low
**Where**: `src/mind_mem/hybrid_recall.py:591-596`
**What**: Loop mutates `r["score"]` directly. The dicts are `.copy()`-ed
from `block_data` in `rrf_fuse` (line 105), but `fused[:limit]` and `fused`
both alias the same objects post-mutation. Inconsistent with the
project-wide immutability convention.
**Fix**: Rebuild list: `[{**r, "score": ..., "_temporal_decay": ...} for r in results]`.

### R-11 — `prefetch_context` category-boost is dead code with multi-signal 🟢 Low
**Where**: `src/mind_mem/_recall_core.py:1326-1338`
**What**: Category hits appended after parallel signal phase, then sliced
off by `results[:limit]`. With 4 signals × `limit=5`, the list reaches 20
before category append, then truncates to 5.
**Fix**: Reserve category slots before appending.

### Confirmed correctness
- ✅ RRF formula correct (`enumerate` 0-based + `rank_0 + 1` = standard
  1-based RRF).
- ✅ Empty `ranked_lists` guard correct.
- ✅ Rerank short-circuits when disabled.
- ✅ No `time.sleep()` in any `async def`; FastAPI routes are sync and
  run in threadpool — correct.
